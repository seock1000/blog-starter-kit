# Next.js 블로그 스타터 킷 헌장

## 핵심 원칙

### I. 컴포넌트 우선 아키텍처
모든 기능은 명확한 경계를 가진 재사용 가능한 React 컴포넌트로 구축되어야 합니다. 컴포넌트는 반드시:
- 명시적인 props 인터페이스를 가진 독립적인 구조
- 런타임 의존성 없이 독립적으로 테스트 가능
- 사용 예제와 함께 문서화
- Server 또는 Client 컴포넌트를 명시적으로 선언 (`"use client"` 지시문 사용)

**근거**: Next.js App Router는 최적의 서버/클라이언트 렌더링을 위해 명확한 컴포넌트 경계를 요구합니다. 재사용 가능한 컴포넌트는 중복을 줄이고 콘텐츠 중심 애플리케이션의 유지보수성을 향상시킵니다.

### II. 타입 안정성 (타협 불가)
TypeScript는 모든 코드에 반드시 사용되어야 합니다. 타입 위반은 빌드 시간 실패를 의미합니다.
- 타입이 없는 서드파티 라이브러리와 인터페이스할 때를 제외하고 `any` 타입 사용 금지
- 모든 props 인터페이스는 명시적으로 정의되어야 함
- 모든 API 응답과 데이터 모델은 TypeScript 인터페이스를 가져야 함
- 타입 오류는 배포를 차단해야 함

**근거**: 정적 사이트 생성은 런타임 오류의 비용을 증폭시킵니다. 타입 안정성은 콘텐츠 구조 불일치와 API 계약 위반을 빌드 시간에 포착합니다.

### III. 성능 우선 설계
기능은 Core Web Vitals와 빌드 성능을 우선시해야 합니다:
- 정적 생성이 서버/클라이언트 렌더링보다 우선되어야 함
- 이미지는 최적화를 위해 Next.js Image 컴포넌트를 사용해야 함
- 컴포넌트는 클라이언트 측 JavaScript를 최소화해야 함
- 100개 미만의 게시물에 대해 빌드 시간은 60초 미만을 유지해야 함
- Lighthouse 성능 점수는 90점 이상이어야 함

**근거**: 블로그의 성공은 검색 가능성(SEO)과 독자 경험에 달려있습니다. 낮은 성능은 두 지표 모두에 직접적인 영향을 미칩니다.

### IV. 콘텐츠 중심 개발
모든 기능은 Markdown 콘텐츠를 진실의 원천으로 취급해야 합니다:
- 콘텐츠 구조 변경은 하위 호환성을 유지해야 함
- 콘텐츠는 렌더링과 독립적으로 검증 가능해야 함
- Frontmatter 스키마 변경은 마이그레이션 경로를 포함해야 함
- 콘텐츠는 애플리케이션 실행 없이 읽을 수 있어야 함

**근거**: 콘텐츠는 코드보다 오래 지속됩니다. 콘텐츠 이식성과 하위 호환성을 보장하면 블로그의 수명이 보호됩니다.

### V. 정적 우선 제공
동적 기능은 런타임 복잡성을 정당화해야 합니다:
- 정적 사이트 생성(SSG)이 기본값
- 점진적 정적 재생성(ISR)은 성능 정당화가 필요
- 서버 사이드 렌더링(SSR)은 명시적 승인이 필요
- 클라이언트 측 데이터 페칭은 대체 정적 콘텐츠를 가져야 함

**근거**: 정적 사이트는 최고의 성능, 안정성 및 호스팅 단순성을 제공합니다. 런타임 복잡성은 실패 모드와 호스팅 비용을 추가합니다.

## 기술 표준

### 필수 스택
- **프레임워크**: Next.js 15+ (App Router)
- **언어**: TypeScript 5+
- **스타일링**: Tailwind CSS 3+ (유틸리티 우선 접근)
- **콘텐츠**: gray-matter frontmatter 파싱을 사용한 Markdown
- **날짜 처리**: date-fns (일관된 포맷팅)
- **패키지 매니저**: npm, yarn, 또는 pnpm (README에 선택 문서화)

### 제약 사항
- 개발에 Node.js 20+ 필요
- 최신 브라우저 타겟팅 (ES2020+, 최근 2개 버전)
- 모바일 우선 반응형 디자인 필수
- 접근성: WCAG 2.1 AA 최소
- 런타임 데이터베이스 의존성 없음

## 개발 워크플로우

### 기능 개발 프로세스
1. **명세**: `/specs/[feature]/spec.md`에 사용자 시나리오와 승인 기준 정의
2. **계획**: `/specs/[feature]/plan.md`에 기술적 접근 방식 문서화
3. **헌장 검사**: 구현 전 모든 원칙과의 정렬 검증
4. **구현**: 점진적 커밋으로 기능 구축
5. **테스트**: 빌드 성공 및 렌더링 출력이 예상과 일치하는지 확인
6. **문서화**: README 및 인라인 문서 업데이트

### 코드 품질 게이트
- TypeScript 컴파일은 오류 없이 통과해야 함
- Next.js 빌드는 성공적으로 완료되어야 함
- Tailwind CSS는 사용되지 않는 클래스가 없어야 함 (purge 설정 사용)
- 개발 모드에서 콘솔 오류/경고 없음
- 모든 새 컴포넌트는 prop 타입 정의를 가져야 함

### 콘텐츠 품질 게이트
- Markdown 파일은 유효한 frontmatter를 가져야 함
- 필수 frontmatter 필드: `title`, `date`, `excerpt`
- 참조된 이미지는 `/public/assets/`에 존재해야 함
- 링크는 유효해야 함 (내부 또는 외부)

## 거버넌스

### 수정 프로세스
헌장 변경은 반드시:
1. 근거와 영향받는 원칙을 문서화
2. 시맨틱 버전 관리에 따라 버전 업데이트:
   - **MAJOR**: 원칙 제거 또는 거버넌스 변경 중단
   - **MINOR**: 새로운 원칙 또는 확장된 지침
   - **PATCH**: 명확화, 오타 수정
3. 변경사항을 반영하도록 모든 템플릿 파일(plan, spec, tasks) 업데이트
4. 해당되는 경우 기존 기능에 대한 마이그레이션 경로 포함

### 버전 관리 정책
- **헌장 버전**: MAJOR.MINOR.PATCH 형식을 따름
- **비준 날짜**: 원본 채택 날짜 (불변)
- **최종 수정 날짜**: 가장 최근 변경 날짜 (ISO 8601 형식)

### 준수 검토
- 모든 기능은 `plan.md`에 헌장 검사를 문서화해야 함
- 위반 사항은 복잡성 추적 테이블에서 정당화되어야 함
- 정당화되지 않은 위반은 기능 승인을 차단함
- 표준 워크플로우를 위해 `.specify/templates/` 사용

### 단순성 강제
복잡성은 정당화되어야 합니다:
- 새로운 npm 의존성 추가는 명시적 근거가 필요
- 새로운 파일 구조 패턴 도입은 문서화가 필요
- Next.js 규칙에서 벗어나는 것은 성능 정당화가 필요
